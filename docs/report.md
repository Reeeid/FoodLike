## 1. 概要

FoodLikeは「グループ外食の店選び」を支援するWebアプリである。最大の特徴は、各メンバーの苦手なもの(ジャンル・食材)を**本人以外には見せないまま**、グループ全員が食べられるお店を提案する点にある。「辛いものが苦手」「エビアレルギー」といった打ち明けにくい情報をシステムが預かり、提案結果にだけ反映することで、気兼ねのない店選びを実現する。

- 開発期間: 【2026年◯月◯日〜7月◯日】
- 開発体制: 【個人開発(+AIペアプログラミング: Claude Code)】
- 公開URL: 【https://xxx.vercel.app】

## 2. 主な機能

| 機能 | 説明 |
|---|---|
| 認証 | Firebase Authentication(Google / メール+パスワード / 匿名ゲスト)。初回ログイン時にバックエンドがメンバーをJIT作成 |
| グループ管理 | 作成・招待コード/QRコードによる参加。QRは`/join?code=xxx`のURLを埋め込み、読み取るだけで参加完了 |
| 好み登録 | プロフィール画面で「苦手なもの」を登録。**他メンバーには一切表示されない** |
| お店提案 | 全員の苦手なものを集約(誰の苦手かは消去)→候補を絞り込み。全員OKが0件なら妥協案をランキング |
| グループチャット | メンバー同士のやり取り(3秒ポーリング)。ホーム画面に未読バッジ(アプリ内通知) |
| AI検索 | チャット内のボタンから自然文で相談すると、おすすめがストリーミング表示され、回答は履歴としてグループ全員に共有される |

## 4. アーキテクチャ

### 4.1 システム構成
【構成図①を挿入】

デプロイは全サービスの無料枠内に収めた(後述のコスト設計)。

### 4.2 レイヤードアーキテクチャ(バックエンド)
【構成図②を挿入】

クリーンアーキテクチャに倣い、`domain → usecase → adapter → infrastructure` の4層構成とし、依存の向きを内側(domain)に固定した。層をまたぐ依存は`port`パッケージのインターフェースで抽象化している。

この設計の効果は「差し替え可能性」に現れている。開発中、以下の実装をすべて**usecase層以下を変更せずに**差し替え(または差し替え予定に)できた:

- メッセージ保存: インメモリ仮実装 → GORM実装
- 店舗データ: モック → ホットペッパーAPI(予定)
- AI応答: ルールベース → LLM API(必要になれば)
- 認証: モック(X-Member-IDヘッダー) ⇔ Firebase(環境変数で起動時切替)

### 4.3 認証フロー
【シーケンス図③を挿入】

認証方式の検証はミドルウェアに集約し、ハンドラは`currentMemberID(c)`経由でしか呼び出し元を知らない。このためローカル開発用のモック認証と本番のFirebase認証を、ハンドラ無変更で切り替えられる。

## 5. 設計上の工夫・意思決定

### 5.1 プライバシー: 好みデータの隔離
「苦手なもの」はこのアプリで最もセンシティブなデータである。以下の多層防御を行った。

1. APIレベル: 好みを返すエンドポイントは`/api/me/preferences`(本人のみ)だけ。グループ経由で他人の好みを取得するAPIは存在しない
2. DTO層: ドメインモデルをそのままJSONにせず、外部に出す形をDTOで明示。Preferenceを含むレスポンス型は本人向け以外に定義していない
3. 提案ロジック: 好みは集約段階で「誰の苦手か」という対応関係を消してから絞り込みに使う(usecaseのコメントにも明記)

### 5.2 認可: BOLA(IDOR)対策
グループIDはURLに現れる連番であり、他人のグループIDを推測してアクセスする攻撃(BOLA)が成立し得る。対策として、**認可チェックをusecase層に置き**、非メンバーには404(存在自体を隠す)を返す方針で統一した。認証(誰か)はミドルウェア、認可(そのリソースに触れるか)はusecaseという役割分担である。

### 5.3 コスト設計: AI機能を無料で提供する
AI検索を素朴に「毎回LLM APIを呼ぶ」設計にすると、利用者数に比例した従量課金が発生し、無料での公開が難しくなる。本アプリでは以下のハイブリッド構成を採用した。

1. 提案・AI検索の中身はルールベース(好み集約→絞り込み→ランキング)で実装し、実行コストをゼロにした
2. AI応答の生成は`AIResponder`インターフェースの背後に隔離した。現在はテンプレート文生成+擬似ストリーミングだが、LLM実装への差し替えは1ファイルで済む
3. AI回答はチャットメッセージとして保存し、グループ全員で共有する。仮に将来LLMを使っても、1回の生成コストをメンバー全員で償却できる構造になっている

試算では、LLM(Claude Haiku 4.5)を使った場合でも1検索あたり約0.6円であり、レート制限(例: 1人1日20回)と組み合わせれば小規模サービスとして十分成立することを確認した上で、まずはゼロコスト構成を選択した。

### 5.4 リアルタイム性: SSE+ポーリングの使い分け
- AI検索の回答: **SSE(Server-Sent Events)**でストリーミング。体感速度が上がり、生成中であることが伝わる。EventSourceはAuthorizationヘッダーを送れないため、fetchのReadableStreamでSSEを自前パースした
- チャットの新着: **3秒間隔のポーリング**。WebSocketより実装・インフラが単純で、Cloud Runのようなサーバーレス環境とも相性が良い。`after_id`による差分取得で転送量を抑えた

### 5.5 招待体験: QRコード
招待コードの手入力は体験が悪いため、`/join?code=xxx`のURLをQR化した。読み取るとログイン済みならワンタップで参加、未ログインならログイン誘導が出る。QR生成はフロントエンドで完結し(ライブラリ`react-qr-code`)、サーバーに画像生成の負荷はない。
